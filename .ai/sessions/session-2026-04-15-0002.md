## 작업 흐름

### 1. 컨플루언스 공유 포맷 문서 작성
사용자가 사내 교육 내부 공유용 컨플루언스에 본 MCP 서버 진행상황을 공유해야 한다고 했다. 기존 `AptLegal MCP`(kor-legal-mcp 기반 법률자문) 페이지 스크린샷 5장(`컨플루언스1~5.png`)을 리포 루트에 올려주고 "동일한 포맷으로"를 요청.

**이미지 판독 기술적 디테일**: Read 도구가 이미지를 자동 다운샘플링해서 원본 해상도로는 텍스트가 읽히지 않았다. 우회 방법:
1. PIL로 이미지 4등분(세로)
2. 각 타일을 2x upscale (Lanczos)
3. 한 번에 하나씩 Read — 배치로 여러 타일을 Read하면 첫 번째만 원본 해상도로 렌더됨

스크린샷 5장 × 타일 4개 = 20개 타일을 순차적으로 Read해서 원본 컨플루언스 페이지의 전체 레이아웃·텍스트를 추출했다.

**작성 결과물**: `confluence_apt_domain_mcp.md` (리포 루트, 커밋 안 함)
- 섹션 1: 개요 (Use Case 명, 서비스 개요, 문제 정의, 목표, 주요 기능 — 8개 tool 전부 분류, 사용자, 기대 효과)
- 섹션 2: 솔루션 아키텍처 — 3-리포/4-계층 구조를 mermaid flowchart로 (카카오톡 → apt-legal-agent → kor-legal-mcp + apt-domain-mcp → PostgreSQL/LiteLLM/Milvus)
- 섹션 3: 서비스 플로우 2종 mermaid
  - (1) 입주민 단지 질의 플로우 — 단톡방-단지 매핑으로 `complex_id` 식별, Router Agent가 Legal/Domain 두 갈래로 분기하여 Synthesizer로 합성
  - (2) 단지 문서 인제스트 플로우 — 파서 → LLM 태거 → Trigger Gate → Postgres → Wiki 생성기
- 섹션 4: Phase 1 진행 현황 체크리스트

포맷은 원본 AptLegal MCP 페이지를 충실히 미러링(테이블 + 불릿 + mermaid 다이어그램)하되, 관점을 본 리포 경계에 맞춰 **apt-domain-mcp 단독**으로 재작성. mermaid 3개는 web version Claude에서 PNG로 별도 렌더해 사용자가 컨플루언스에 붙이는 흐름으로 합의.

**컨플루언스용 산출물은 리포에 커밋하지 않음** (교육 자료, git 추적 대상 아님). 파일은 working tree에만 남아있음.
