# Session 2026-04-14-0001

첫 번째 세션. `apt-domain-mcp` 리포 자체가 이 세션에서 생성되었다.

## 프로젝트 개요
`apt-domain-mcp` — 공동주택 **단지별 도메인 지식**을 제공하는 MCP 서버.
관리규약(조문 단위), 입주자대표회의 회의록, 운영 산출물(감사·공고·장기수선)을 **단지(`complex_id`) 단위**로 색인한다. 법령·판례·자치법규 일반 조회는 sibling 리포 `kor-legal-mcp`의 책임이며 본 서버는 그 데이터를 직접 쿼리하지 않는다.

파이프라인 내 위치:
```
ChatGPT Enterprise CustomGPT
       ↓ (A2A)
Vertical AI Agent (apt-legal-agent, 예정)
       ↓ (MCP)
   ┌───┴────────────────────────┐
   ↓                            ↓
kor-legal-mcp              apt-domain-mcp  ← 본 리포
(법령·판례·자치법규)         (단지별 규약·회의록·위키)
   ↓                            ↓
law.go.kr Open API         PostgreSQL (+ Milvus)
```

## 세션 배경
직전 `kor-legal-mcp` 세션(2026-04-14-0003)에서 15개 law.go.kr tool 확장과 CustomGPT 연결이 끝났고, 다음 단계로 "apt-domain-mcp 신규 리포 설계" 항목이 남아 있었다. 이번 세션은 그 후속으로 시작해, 지역/단지별 도메인 지식 구성 방법 논의 → 데이터 소스 조사 → 가상 단지 기반 Phase 0 스캐폴드 완성까지 수행했다.

## 최근 작업 내역

### 1. 데이터 소스 지도 작성 (K-apt 등)
`apt-domain-mcp`의 범위를 정하기 위해 실제 공개 데이터 소스를 웹에서 검증.

- **L2 시·도 관리규약 준칙**: `myapt.molit.go.kr/board/398.do` (중앙공동주택관리지원센터, 국토부 직영)에 전국 17개 시·도 준칙 집약. AJAX 로딩이라 크롤링에는 DevTools로 list API 파악 필요.
- **L3 단지별 관리규약**: K-apt OpenAPI에는 **관리규약 전문 엔드포인트 없음**. K-apt 웹사이트에 형식상 탭은 있으나 파일럿 단지(부천시 보람마을 아주아파트)에 대해 **실데이터가 비어 있음 확인**. → L3는 K-apt에 의존할 수 없고 단지 업로드 파이프라인 only로 설계.
- **L4 운영 산출물 (관리비·입찰·감사)**: K-apt OpenAPI 일부 존재(`AptBasisInfoServiceV4`, 입찰공고 서비스 등). 부천시 공동주택나눔터(`apt.bucheon.go.kr`)는 단지정보/입찰정보/수의계약정보/관리비정보/주요통계 5개 카테고리만 있고 관리규약·회의록·감사보고서는 없음.
- **결론**: 파일럿 단지의 1,2,3번(관리규약 전문·개정이력·회의록)은 **입대의에 직접 문의**해서 받는 방식으로 실제 데이터 수급. 공개 포털 의존을 포기하니 오히려 아키텍처가 단순해짐 — 인제스트 파이프라인이 "단지 문서 업로드 → 파싱 → 인덱싱" 한 경로로 수렴.

### 2. 아키텍처 결정: 3계층 저장소 (벡터 DB는 회의록에만)
사용자가 AX Advanced 교육에서 얻은 통찰("실제 현장에서 Vector DB는 큰 의미가 없고 메타데이터 풍부화가 더 중요")을 토대로 Vector 남용을 피하는 방향으로 설계.

- **L1 PostgreSQL (코어)**: 관리규약 조문 단위 정규화, 개정 이력 append-only, FTS(tsvector + 한국어 analyzer 선택 보류: pg_bigm vs mecab-ko). 카테고리·태그는 **인제스트 타임에 LLM이 미리 태깅**해 `TEXT[]` 컬럼으로 저장 → 조회 타임 LLM 0회, RDB 필터로 답한다.
- **L2 LLM Wiki (PostgreSQL row + MCP resource)**: 토픽별("주차", "관리비", "반려동물", "층간소음" 등) 큐레이션 페이지를 LLM이 소스에서 합성. 파일 시스템 아닌 `wiki_page` row로 저장, `resource://complex/{id}/wiki/{topic}` URI로 노출. 소스 해시 변동 시 재생성 큐.
- **L3 Milvus (회의록 전용)**: 관리규약은 영구 벡터 제외(구조가 있으니 낭비). 회의록 자유서술부에만 적용. 배포 타깃 AWS EKS에 PostgreSQL·Milvus 모두 기존 제공되므로 "인프라 이슈가 아닌 코퍼스 적합성 이슈"로 판단을 명문화. Phase 1 후반 FTS 측정 후 필요성 확인되면 즉시 도입.

## 중요 참고사항

### 리포 동기화 패턴 (kor-legal-mcp와 유사하지만 주의)
- **github (`origin`)**: `https://github.com/youngs75/apt-domain-mcp.git` — source of truth
- **포털 GitLab**: 배포 타깃. **`main` 브랜치 protected, force push 불가**. 사용자가 직접 unprotect도 불가.
- **sync 패턴**: kor-legal-mcp는 `git fetch github && git reset --hard github/main && git push origin main` 패턴이지만, **본 리포는 force 불가이므로 merge 방식**으로만 가능:
  ```
  git fetch github
  git merge github/main --no-ff -m "chore: sync github/main"
  git push origin main
  ```
- `.devcontainer`는 이제 github에도 포함되어 있어서 추가 유지 로직 불필요. 단 github main = GitLab main이 아닐 경우 merge commit이 GitLab 쪽에 쌓임 — 감안하고 갈 것.

### 배포 트러블슈팅 누적 (본 세션 교훈)
- `pyproject.toml`에 `readme` 필드가 있으면 Dockerfile에 `COPY README.md ./`가 필수. 없으면 빌드 실패. 이번엔 필드를 제거하는 쪽으로 해결. kor-legal-mcp와 동일.
- 포털 빌드 config가 Dockerfile path를 루트로 보도록 되어 있어야 함. 서브디렉토리 설정이면 `Dockerfile not found`.
- MCP Inspector는 endpoint URL에 반드시 **`/mcp` suffix** 포함 (kor-legal-mcp 교훈 승계).

### 코드/빌드 함정 (kor-legal-mcp에서 승계)
- `TransportSecuritySettings(enable_dns_rebinding_protection=False)` 필수 (프록시 뒤 421 방지)
- `FastMCP`의 `streamable_http_app()`은 자체 `/mcp` 라우트 보유 → `Mount("/", ...)` 사용 (서브 경로 금지, 307 redirect 방지)
- `streamable_http_app()`의 lifespan은 outer lifespan에서 delegate 필수 (`lifespan_context`) — 누락 시 "Task group is not initialized"
- `MCP tool 응답은 원문 그대로`, truncate 금지 (kor-legal-mcp 원칙 승계)

### 설계 원칙 (반드시 지킬 것)
- **단지 격리**: 모든 tool의 첫 파라미터는 `complex_id`. tool handler 레벨에서 누락 시 즉시 `INVALID_PARAMS`. 한 단지의 쿼리가 다른 단지 데이터를 보는 일은 절대 없어야 함.
- **메타데이터 우선주의**: 벡터 유사도에 의존하지 말고 인제스트 타임 LLM 태깅 + RDB 필터. 관리규약은 영구적으로 벡터 인덱스 대상 아님.
- **append-only 버전**: `regulation_article`은 update가 아니라 새 version row. `is_current=true`는 partial unique index로 단일 보장.
- **원문 우선**: 요약·해석은 상위 LLM 책임. tool은 원문만 반환.
- **도메인 vs 법령 경계**: "공동주택관리법 제X조"는 `kor-legal-mcp`, "우리 단지 관리규약 제X조"는 `apt-domain-mcp`. 상위 Agent가 질의 분해·라우팅.

### 환경 / 인프라
- **로컬**: Citrix VDI Windows 11, PowerShell 주 사용. Python 3.14 (`.venv\Scripts\python.exe`). Docker/WSL2 불가 — Docker 빌드는 포털 Web IDE only.
- **Bash tool**: Git Bash 환경. `uv` 바이너리는 `/c/Python314/Scripts/uv`로 접근 가능(이번 세션에서 `uv lock`을 Bash에서 직접 실행 성공). 단 메모리에 저장된 원칙대로 Python/uv 장기 실행은 여전히 사용자 PowerShell로 delegate하는 게 안전.
- **배포 환경 (AWS EKS)**: PostgreSQL·Milvus 모두 포털 별도 서비스로 기존 제공. 벡터 도입은 인프라가 아닌 코퍼스 적합성 판단에만 의존.
- **프록시 뒤**: kor-legal-mcp와 동일 환경. `TransportSecuritySettings(enable_dns_rebinding_protection=False)` 필수.
