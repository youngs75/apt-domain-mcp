### 2. DB 접속 진단 (잠깐의 우회로)

사용자가 생성한 RDS 정보를 `.env`에 `DATABASE_URL` 형태로 저장 (gitignored 확인). 로컬에서 asyncpg 직접 접속 시도 → 10초 타임아웃. SSL/TLS 옵션을 끄거나 permissive로 바꿔도 동일. 사용자가 "프록시 때문 아닌가"라고 추정.

원시 socket 진단으로 결정타 확인:
- DNS: `mspsapoc-postgresql.cilccw66wsss.us-east-1.rds.amazonaws.com → 35.170.150.84` 정상
- TCP:5432 RDS: 타임아웃
- TCP:443 google.com (직접): **타임아웃** ← 결정타

VDI는 HTTP 프록시(70.10.15.10:8080) 거친 80/443 HTTP/HTTPS만 인터넷 egress 허용. raw TCP는 어떤 포트도 막힘. 즉 RDS가 public이든 private이든 로컬에서 접근 불가. 포털 Web IDE에서 실행하는 경로로 전환. 이 사실은 향후 모든 인프라 작업의 전제이므로 메모리에도 갱신 필요.

→ **메모리 업데이트 대상**: VDI 환경에서 raw TCP egress가 막혀 있어 외부 PostgreSQL 같은 서비스는 로컬에서 절대 접속 불가. 모든 DB/벡터스토어 작업은 포털 Web IDE에서 실행해야 함.

### 3. 인제스트 파이프라인 (`src/apt_domain_mcp/ingest/`)

**구조**
- `models.py` — `ParsedArticle`, `ParsedRegulation`, `ParsedRevisionEntry`, `ParsedRegulationDiff`, `ParsedDecision`, `ParsedMeeting` (plain dataclass, in-process only)
- `parser_regulation.py` — `## 제N장`, `### 제N조 (제목)` 패턴 기반 state machine 파서. 84조 정상 파싱
- `parser_regulation_diff.py` — `## N. 제XX조 (title) — 개정/신설/삭제` entry + `### 현행/개정/신설 조문/개정 사유/신설 사유` subsection 파서. **`apply_diff_to_articles(base, diff)` 헬퍼**가 핵심: v2/v3 row를 만들 때 v1 base에 diff를 얹어 full snapshot을 만든다 → regulation_article 테이블에 모든 버전이 전체 조문을 가지도록 함
- `parser_meeting.py` — meta 헤더(일시·유형·참석·정족수) + `## 안건 N.` 블록 + `### 결정사항` 서브섹션 파서. 결정사항 라인에서 결과(가결/부결/보류), 투표(찬성/반대/기권), 관련 조문, 후속조치 추출
- `tagging.py` — **키워드 기반 카테고리 태거 (LLM 스텁)**. 17개 카테고리(주차·관리비·반려동물·층간소음 등)의 키워드 룰. 본문에서 조문 참조 추출 정규식, 법령 참조 추출 정규식. 실제 LLM 호출은 Phase 1 후반으로 이월
- `repository.py` — asyncpg 함수들. `upsert_complex`, `upsert_document`(sha256 dedup), `upsert_regulation_version`(make_current 시 기존 row를 false로 일괄 업데이트 → unique partial index 유지), `upsert_regulation_articles`(version 단위 전량 교체), `upsert_regulation_diff`(revision 전량 교체), `upsert_meeting`(date+type 기준 dedup), `_insert_decision`. ULID는 외부 의존성 없이 timestamp+random 26-char 생성기 자체 구현
- `seed.py` — 합성 단지 end-to-end 시드. 트랜잭션 1개 안에서: complex → v1 → v2(diff적용) → v3(diff적용, current) → meetings 3건. 멱등(재실행 OK)
- `cli.py` — `python -m apt_domain_mcp.ingest.cli --complex-id ... --kind regulation|regulation-diff|meeting --file ...`. 운영에서 단일 파일 인제스트할 때 사용

**파서 디버깅**
- v1 84조 정상 파싱, 제13/39/41/55조 모두 추출
- v2 diff 첫 시도에서 **regex alternation 버그 발견**: `(현행|개정|...|개정 사유|...)` 순서가 잘못되어 "### 개정 사유" 헤더가 "개정"으로 매치돼 reason이 빈 문자열로 들어옴. `(개정 사유|신설 사유|삭제 사유|신설 조문|현행|개정)` 순서로 재정렬 (긴 alternative 먼저). 재테스트 후 v2/v3 diff 모두 old/new/reason 정상 채워짐.
- 회의록 3건 모두 안건·투표·관련 조문 정상 추출. category 태깅도 합리적으로 붙음 (장기수선·관리비·시설 등)

### 4. DB 기반 MCP tool 8종

**구조**
- `src/apt_domain_mcp/db/__init__.py` — 공유 asyncpg pool. `init_pool()` (DATABASE_URL 없으면 None 유지), `acquire()` async context manager
- `src/apt_domain_mcp/tools/handlers.py` — 8개 핸들러. 모두 `complex_id` 첫 파라미터, `_require_complex_id()`로 빈값 체크 + `_ensure_complex(conn, complex_id)`로 존재 확인 → 단지 격리 강제. 에러 코드 통일: `DB_NOT_CONFIGURED`/`INVALID_PARAMS`/`COMPLEX_NOT_FOUND`/`ARTICLE_NOT_FOUND`/`MEETING_NOT_FOUND`/`WIKI_NOT_FOUND`
- `server.py` 전면 교체 — Phase 0 placeholder 폐기, lifespan에서 `db.init_pool()` 호출 후 inner MCP lifespan 위임 (kor-legal-mcp 패턴 유지). `/healthz`가 실제 PG ping 수행

**Tool 명세**
1. `list_complexes()` — 단지 메타 row 전체
2. `get_complex_info(complex_id)` — 단지 상세 + 현행 regulation 버전 + meeting_count 집계
3. `search_regulation(complex_id, query, category=None, version=None, limit=10)` — body/title ILIKE, version 미지정 시 is_current. category는 ANY() 매치
4. `get_regulation_article(complex_id, article_number, version=None, include_history=True)` — 조문 단건 + revision_history(`regulation_revision` JOIN with effective_date)
5. `list_regulation_revisions(complex_id)` — versions 목록 + revisions 목록 (장 단위 변경 이력)
6. `search_meeting_decisions(complex_id, query=None, category=None, result=None, date_from=None, date_to=None, limit=20)` — 동적 SQL 빌더. 결정문/안건/후속조치 ILIKE + category ANY + result 등치 + 날짜 범위
7. `get_meeting_detail(complex_id, meeting_id)` — 회의 메타 + 안건 전체 + raw_text
8. `get_wiki_page(complex_id, topic)` — `wiki_page` 조회. Phase 1 후반까지는 `WIKI_NOT_FOUND` 정상

**원칙**
- truncate 금지(원문 우선) — 조문 본문, 회의록 raw_text 그대로 반환
- `ensure_ascii=False` — 한글 그대로 (kor-legal-mcp 승계)
- 단지 격리는 SQL WHERE에서 강제 (애플리케이션 레벨 신뢰 X)

## 중요 참고사항

### VDI 네트워크 제약 (이번 세션 새로 확정)
- VDI에서 raw TCP egress는 모든 포트 차단. HTTP 프록시(70.10.15.10:8080) 거친 80/443 HTTP/HTTPS만 인터넷 접근 가능
- 따라서 PostgreSQL/Milvus 같은 외부 서비스는 **로컬에서 절대 접속 불가**. 모든 DB 작업은 포털 Web IDE에서 실행
- 원시 socket 테스트로 확정 (TCP:5432 RDS 타임아웃, TCP:443 google.com 직접도 타임아웃, DNS만 정상)
- 외부 HTTPS API(MCP endpoint 등)는 curl/httpx로 가능 (proxy via CONNECT 터널). 본 세션의 외부 스모크 테스트가 그 경로

### 포털 동기화 패턴 (직전 세션 승계 + 보강)
- github (`origin`): source of truth
- 포털 GitLab `main`: protected, force push 불가 → merge 방식 sync only
- 본 세션은 사용자가 포털 Web IDE에서 `git merge --no-ff github/main` 수행. 결과 GitLab 헤드는 merge 커밋 1개 더 얹힌 형태로 수렴
- Web IDE의 클로드 코드에게 작업을 위임할 때는 **자율 진행 가능한 단계별 프롬프트**를 만들어 그대로 복사·붙여넣기. 본 세션의 7단계 프롬프트는 잘 동작했음

### 외부 endpoint 노출 (포털 운영 메모)
- 포털 빌드 + 배포 + **외부 endpoint 등록**이 별개 단계임을 확인
- endpoint 미등록 상태에서는 외부에서 AWS ELB 503 (`Server: awselb/2.0`) 응답. /healthz까지 503으로 보임
- 등록 후 새 endpoint URL이 생성되며, 기존 URL과 다를 수 있음 (이번에 `portal-serving-mcp-a3f7b6e6c62d` → `portal-serving-evangelist-1-mcp2-c01386b6`로 변경)
- 검증: `/healthz`가 200 + `postgres: ok` 나오면 외부 노출까지 정상

### MCP Streamable HTTP 클라이언트 패턴 (httpx)
이번 세션 외부 스모크 테스트로 검증된 흐름:
```python
headers = {
    "Accept": "application/json, text/event-stream",
    "Content-Type": "application/json",
}
# 1. initialize (no session id yet)
r = c.post(URL, json={"jsonrpc":"2.0","id":1,"method":"initialize",
    "params":{"protocolVersion":"2024-11-05","capabilities":{},
              "clientInfo":{"name":"smoke","version":"0.1"}}}, headers=headers)
sid = r.headers.get("mcp-session-id")
headers["Mcp-Session-Id"] = sid
# 2. notifications/initialized
c.post(URL, json={"jsonrpc":"2.0","method":"notifications/initialized"}, headers=headers)
# 3. tools/list, tools/call ...
```
응답은 SSE 라인(`data: {...}`) 형태로 올 수 있어서 parse helper 필요.
verify=False는 사내 프록시 TLS 가로채기 환경에서만. 포털 외부 직접 접근 환경에서는 verify=True.

### 합성 데이터 매핑 결정 (이번 세션 확정)
spec과 v1.md 조문 번호가 어긋났던 부분은 v1.md를 source of truth로 두는 방향으로 정리. 향후 합성 데이터 추가 시에도 동일 원칙: **이미 합성된 산출물(특히 본문이 큰 v1.md)을 먼저 source of truth로 두고 spec을 거기에 맞춰 재해석**한다. spec 자체는 의도였고, 실 산출물이 권위.

신설 조문 번호 규칙: 기존 조문 번호 끝에 새 번호를 추가하는 방식 (제85조, 제86조). "조의2" 형태(제68조의2) 도입은 회피 — 파서 복잡도 증가, 사용자 가독성 저하.

### tagging 과다 매칭 관찰 (Phase 1 후반 LLM 교체 근거)
키워드 룰 기반 tagger가 본문 내 부수적 단어까지 잡아 over-match 하는 경향:
- 제13조 (구성): cats=['대표회의', '반려동물', '선거'] — '반려동물'은 본문에 없는데 카테고리 키워드 매칭 오류로 보임. 추적 필요
- 제85조 (반려동물): cats=['관리비', '대표회의', '관리주체', '시설', '주차', '반려동물', '선거'] — 본문에 "입주자대표회의의 의결로 정한다", "관리주체는 게시판에 공고한다" 같은 일반 행정 문구가 들어가서 카테고리가 폭증
- 임팩트는 낮음(검색 결과의 false positive로 한두 건 더 잡히는 정도). LLM 교체 시 자연스럽게 해소
- 즉시 fix가 필요하면 카테고리 키워드를 더 좁히거나 본문에 카테고리 명시 키워드가 1회 이상 나오는 경우만 매칭하는 식으로 룰 강화 가능. 다만 LLM 교체 예정이라 fast forward로 진행

### 코드/빌드 함정 (kor-legal-mcp + 0001 세션 누적)
- `TransportSecuritySettings(enable_dns_rebinding_protection=False)` 필수
- `streamable_http_app()`은 자체 `/mcp` 라우트 보유 → `Mount("/", ...)` 필수, 서브 경로 금지
- inner lifespan을 outer lifespan에서 위임 필수 (`router.lifespan_context`)
- `pyproject.toml`에 `readme` 필드 두면 Dockerfile에 README.md COPY 필요. 없으면 빌드 실패. 본 리포는 필드 제거 상태
- 본 세션 추가: pg_trgm 확장이 없는 환경에서도 `gin_trgm_ops` 인덱스 부분이 NOTICE/SKIP으로 처리되고 스키마 적용은 성공. ILIKE는 trgm 없이도 동작. `CREATE EXTENSION IF NOT EXISTS pg_trgm`은 권한이 없으면 조용히 패스(드라이버 환경에 따라 다름)

### 설계 원칙 (반드시 유지)
- 단지 격리: 모든 tool 첫 파라미터 `complex_id`, SQL WHERE에서 강제, `_ensure_complex` 헬퍼로 존재 확인
- 메타데이터 우선주의: 벡터 의존 X, 인제스트 타임 태깅 + RDB 필터로 응답
- append-only 버전: regulation_article은 version 단위 row, `is_current=true`는 partial unique index로 단일 보장
- 원문 우선: tool 응답에서 truncate 금지
- 도메인 vs 법령 경계: "공동주택관리법 제X조"는 kor-legal-mcp, "우리 단지 제X조"는 apt-domain-mcp
