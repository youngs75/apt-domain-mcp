### 6. 배포 메커니즘 혼동 및 FastMCP 래퍼 버그 발견
세션 후반 "배포 타이밍 맞느냐"는 사용자 질문에 내가 "매 커밋마다 포털 자동 배포 중"이라고 잘못 답변. 사용자가 "GitLab push만으로는 배포되지 않는다" "포털에서 수동 배포가 필요하다"고 정정.

**진상**:
- 이 세션의 커밋 대부분(태거, Wiki CLI, timeout fix)은 **MCP 서버 런타임과 무관한 CLI 코드**라 재배포 불필요 (Web IDE에서 `seed.py`/`wiki/cli.py` 직접 실행하면 DB만 업데이트됨)
- 유일하게 재배포가 필요했던 `tools/handlers.py`의 `search_regulation` query optional 변경은 **실제로는 한 번도 라이브에 반영된 적 없었음**

**결정적 버그**: 내가 `handlers.py`의 내부 함수 시그니처만 optional로 바꾸고, `server.py`의 **FastMCP 래퍼** `_search_regulation`은 놓쳤음. FastMCP는 래퍼 함수 시그니처에서 Pydantic input model을 생성하므로, MCP 클라이언트가 보는 스키마는 래퍼 쪽이다. 내 수정은 코드 레벨에서 의미 없었고, 외부 `search_regulation(category="반려동물")` query 생략 호출은 **"Field required" 422**를 계속 반환했다.

**수정** `18cd5ea`: `server.py`의 `_search_regulation` 래퍼도 `query: str | None = None`으로 교체 + description 갱신 ("query 또는 category 중 최소 하나")

**재배포 + 재검증**:
- 사용자가 포털 UI에서 수동 배포
- 로컬 VDI에서 httpx로 직접 스모크 (HTTPS 프록시 경유 가능):
  - `/healthz`: 200, `version=0.1.0 phase=1 postgres=ok`
  - `search_regulation(category="반려동물")` query 생략: **count=2** (제59조·제85조) ✅
  - `search_regulation(category="대표회의")` query 생략: count=20, 태거 정확도 재확인
  - `search_regulation(no query, no category)`: `INVALID_PARAMS` 정상 거부
  - `search_meeting_decisions(category="층간소음")`: 1건 (2025-05-30 제86조 최초 적용)
- 메모리 `infra_portal_deploy.md` 저장 (GitLab push ≠ 배포, 수동 트리거 필요)

## 중요 참고사항 (메모리에 별도 저장되지 않은 것)

### FastMCP 래퍼와 핸들러 동기화 원칙 (이 세션 하드 러닝)
FastMCP tool의 input schema는 `@mcp.tool` 데코레이터가 붙은 **래퍼 함수의 시그니처**에서 Pydantic이 자동 생성한다. 내부 핸들러(`handlers.py`)의 시그니처를 바꿔도 래퍼가 그대로면 클라이언트 측 스키마는 바뀌지 않는다.

**규칙**:
- 핸들러 시그니처와 래퍼 시그니처는 반드시 1:1 매칭
- 핸들러 변경 시 래퍼도 동시 수정 필수
- `tests/test_mcp_tool_schemas.py`가 이 원칙의 방어선. 수정 시 테스트 업데이트 또는 스키마 회귀 확인

### 배포 메커니즘 (이 세션 새로 확정)
- **GitLab push ≠ 배포**. 배포는 **사용자가 포털 UI에서 수동 트리거**해야 함
- 재배포 필요한 변경은 런타임 코드(`server.py`, `tools/handlers.py`, `db/`, `tools/`) 변경에 한정
- 다음은 재배포 **불필요**: `ingest/*` CLI, `wiki/cli.py`, `tagging.py`, `llm_client.py` (이것들은 Web IDE에서 직접 실행하면 DB만 업데이트)
- Web IDE 위임 프롬프트에서 런타임 코드 변경 시 반드시 "수동 재배포 필요" 단계 명시할 것
- 메모리 `infra_portal_deploy.md` 참조

### LiteLLM + Bedrock Claude Known Quirks
1. **`response_format={"type":"json_object"}` 미지원** — 전달 시 content가 빈 `"{}"`만 반환. 파라미터 제거 + 프롬프트에 "JSON만 응답" 지시 + fence 제거 파서로 우회. `LITELLM_USE_JSON_MODE=0` 기본값
2. **기본 timeout 30초로는 긴 프롬프트 처리 부족** — Wiki 생성처럼 큰 프롬프트는 120초+필요. `LITELLM_TIMEOUT=120` 기본값, APITimeoutError 시 1회 재시도(300s)
- 메모리 `infra_litellm_bedrock_json_mode.md` 참조. apt-legal-agent 쪽 LLM 호출에서도 동일 주의 필요
